### Краткое описание проекта:
---

WEB-приложение, предназначенное для проведения онлайн-тестирований преподавателями высших и средних учебных заведений.

##### Особенности:

- Четкая система ролей с разным уровнем доступа
- Возможность добавлять соавторов для редактирования тестов и сохранение истории изменений
- Автоматическая проверка тестов и выставление оценок в 100-бальной и 5-бальной системе 
- Удобный функционал для обновления данных, забирающий на себя часть монотонной работы  
- Возможность расширения и интеграции со сторонними сервисами 

##### Используемый стек:

- Java 21
- Maven
- Spring Boot (WEB, MVC, DATA JPA, SECURITY)
- Postgre SQL

### Архитектура:
---

Приложение реализовано, как модульный монолит. Благодаря этой структуре можно максимально "безболезненно" добавить дополнительный необходимый компонент или разбить систему на отдельные сервисы.

Всего есть 3 бизнес-модуля и один gateway для унифицированного доступа и авторизации/аутентификации:

Все бизнес-модули имеют имеют гексагональную архитектуру ( или же по другому "Порты и адаптеры"). Доменные модели связаны друг с другом через id, а не ссылаются напрямую (исключение - enumы). Таким образом бизнес-модули работают независимо друг от друга даже внутри монолита и их спокойно можно дополнять необходимыми компонентами

<img width="2141" height="1661" alt="Декомпозиция предметной области drawio" src="https://github.com/user-attachments/assets/f73f7282-05d3-46f2-9294-633bf4d4bc2b" />


Доменные модели лишь описывают предметную область, с которой работает программа. Бизнес-логика зашита в первичных портах, именно они могут всячески преобразовывать данные. Некоторые из них в последствии были разделены на несколько портов, дабы избежать перегрузки

Вторичные же порты на первом этапе обладали лишь базовыми CRUD-операциями для разных объектов предметной области, но вскоре некоторые из них были расширены для оптимизации работы программы

За реализацию первичных портов отвечают сервисы-оркестраторы. Главным императивом при их написании было - "Не изобретай велосипед". По этой причине некоторые реализации переиспользуют свои же методы или зависят от других первичных портов модуля. В данном случае это хорошая практика, так-как увеличивается сцепленность между компонентами и исчезает необходимость дублировать код. При этом сами порты в любом случае остаются независимыми

В прочем не всегда переиспользование было выгодным или вообще представлялось возможным, поэтому можно увидеть сервисы, использующие как вторичные, так и другие первичные порты 

За реализацию вторичных портов ответственны вторичные адаптеры. При использовании Spring Boot они используют JPA репозитории для чтения и изменения данных в SQL-базе данных, но по необходимости их можно переписать для любой формы хранения данных

Контроллеры выступают первичными адаптерами. Они вызывают методы первичных портов. Если контроллеру необходимо несколько методов, то используется "фасад", который вызывает методы в нужном порядке

Маппинг в репозитарную сущность или DTO, а так-же в обратную сторону происходит в адаптерах

Все наполнение модулей можно поделить на Core и Infrastructure. В первой папке находятся доменные модели, первичные и их реализации, а так-же вторичные порты. Вторая папка содержит все остальное

##### Ответственности модулей:

- gateway - набор контроллеров, маршрутизация и настройка правил безопасности
- control_module - управление учетными записями пользователей: регистрация, удаление, чтение и изменение информации. 
- creation_test_module - управление тестами: создание/удаление тестов, добавление версий тестов. Проведение тестирований, изменение правил прохождения теста
- testing_attemts_module - доступ к истории прохождений и задание правил прохождения, связанных с ней
